# SonarQube Unity + LLM Dialogue Playbook

This playbook is the project-specific SonarQube setup for `DevProject`.

Current local stack checked on 2026-04-03:
- SonarQube Server: `26.3.0.120487`
- SonarScanner for .NET: `11.2.1`
- .NET SDK: `10.0.201`
- Unity: `6000.4.0b11`
- Unity Test Framework: `1.6.0`
- Unity Code Coverage: `1.3.0`

## Why this setup exists

The old Sonar flow in this repo started a scanner session and then skipped the build.
That works against the official Sonar guidance for .NET projects and prevents Roslyn-based C# issues from reaching SonarQube.

The current setup fixes that by:
- Running the supported `begin -> build -> end` flow.
- Excluding non-`Network_Game*` assemblies from Sonar during Unity-generated builds.
- Importing Unity's existing generic coverage report from `CodeCoverage/Report/SonarQube.xml`.
- Keeping the Supabase TypeScript LLM proxy inside the same Sonar project.

Official references used for this setup:
- SonarScanner for .NET: https://docs.sonarsource.com/sonarqube-server/10.1/analyzing-source-code/scanners/sonarscanner-for-dotnet/
- Configuring the scanner: https://docs.sonarsource.com/sonarqube-server/2025.1/analyzing-source-code/scanners/dotnet/configuring
- Specifying test projects: https://docs.sonarsource.com/sonarqube-server/2025.2/analyzing-source-code/dotnet-environments/specify-test-project-analysis
- .NET test coverage: https://docs.sonarsource.com/sonarqube-server/2025.3/analyzing-source-code/test-coverage/dotnet-test-coverage
- Advanced exclusions: https://docs.sonarsource.com/sonarqube-server/project-administration/adjusting-analysis/setting-analysis-scope/advanced-exclusions

## Repo policy

Analyze:
- `Assets/Network_Game/**/*.cs` through Unity-generated `Network_Game*` projects.
- `supabase/functions/**/*.ts` through multi-language analysis.

Do not analyze:
- Unity generated assets and metadata.
- Third-party gameplay assets under `Assets/Network_Game/ParticlePack/**`.
- Unity starter assets under `Assets/StarterAssets/**`.
- Repo infrastructure folders such as `.vscode`, `Docs`, `Tools`, `scripts`, and `CodeCoverage`.

Coverage source:
- Use Unity's generic SonarQube report at `CodeCoverage/Report/SonarQube.xml`.
- Do not try to replace Unity editor tests with `dotnet test`; this repo already has the correct Unity coverage pipeline.

## Recommended SonarQube UI customizations

### 1. Create a custom C# quality profile

In SonarQube:
1. Go to `Quality Standards Administration > Quality Profiles`.
2. Copy `Sonar way` for `C#`.
3. Name it `Unity 6 Multiplayer + Dialogue`.
4. Set it as the project profile for `unity-multiplayer-game`.

What to keep strict:
- Cognitive complexity and method size on dialogue routing code.
- Null-safety and defensive checks around auth, ownership, and network state.
- Async/task misuse around request dispatch, cancellation, and background workers.
- String/regex/JSON handling rules for prompt assembly, effect parsing, and LLM response handling.

What not to weaken globally:
- Complexity rules on `NetworkDialogueService`.
- Reliability rules on `LocalPlayerAuthService`, `PlayerBootstrap`, `NetworkBootstrap`, and `NetworkDialogueAuthGate`.

Use those rules as refactor pressure, not noise suppression.

### 2. Create a custom TypeScript profile for the Supabase LLM proxy

Copy `Sonar way` for `TypeScript` and assign it to this project.

Keep strict on:
- Cognitive complexity.
- Explicit error handling and promise chains.
- Input validation around model/provider requests.
- Logging and secret-handling hygiene.

Primary target file today:
- `supabase/functions/llm-proxy/index.ts`

### 3. Set new-code policy

Use a reference branch if your default branch is stable.

Recommended:
- `Project Settings > New Code`
- Set `Reference branch = main` if `main` is your long-lived integration branch.

If you do not yet have a stable mainline, use a version baseline after the next clean full analysis.

### 4. Use advanced exclusions only for real false positives

Use `Project Settings > General Settings > Analysis Scope` for narrow path/rule exceptions.

Good candidates for exclusions:
- `Assets/Network_Game/ParticlePack/**`
- `Assets/StarterAssets/**`
- Any future generated code marked with `// <auto-generated>` or `[GeneratedCode]`

Bad candidates for exclusions:
- `Assets/Network_Game/Dialogue/NetworkDialogueService.cs`
- `Assets/Network_Game/Auth/LocalPlayerAuthService.cs`
- `Assets/Network_Game/Behavior/PlayerBootstrap.cs`
- `Assets/Network_Game/Dialogue/Effects/*.cs`

If a specific rule is noisy in a narrow Unity-only pattern, exclude that rule by path. Do not weaken the entire profile.

1. `Assets/Network_Game/Dialogue/NetworkDialogueService.cs`
2. `Assets/Network_Game/Dialogue/OpenAIChatClient.cs`
3. `Assets/Network_Game/Dialogue/NetworkDialogueAuthGate.cs`
4. `Assets/Network_Game/Dialogue/Effects/ParticleParameterExtractor.cs`
5. `Assets/Network_Game/Dialogue/Effects/EffectParser.cs`
6. `Assets/Network_Game/Auth/LocalPlayerAuthService.cs`
7. `Assets/Network_Game/Behavior/PlayerBootstrap.cs`
8. `Assets/Network_Game/Behavior/NetworkBootstrap.cs`

These files matter most for:
- Prompt-context integrity
- Auth-gated dialogue admission
- Participant routing
- Effect parsing and bounds enforcement
- Network ownership correctness
- Backend fault isolation

## Coverage stance for this repo

Treat coverage as a design signal, not a vanity number.

High-priority coverage targets:
- Auth state transitions
- Dialogue request admission and rejection
- Conversation key generation
- Identity propagation
- Effect parsing and clamp behavior
- LLM action response parsing
- Supabase LLM proxy request validation

Lower-priority coverage targets:
- Visual-only helper scripts
- Third-party code
- Pure scene/demo glue

## Local workflow

1. Refresh Unity coverage when tests change.
2. Run `.\Tools\CodeAnalysis\setup-sonarqube.ps1 -SetupProject`.
3. Run `.\Tools\CodeAnalysis\setup-sonarqube.ps1 -RunAnalysis`.
4. Use SonarQube for project-wide trends and rule tuning.
5. Use SonarQube for IDE for line-level feedback while editing.
